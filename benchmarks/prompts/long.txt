Section one. Background on local large-language-model inference.

For most of the past decade, running a sufficiently capable language model
required either a beefy workstation with a discrete graphics card or a paid
API to a hosted model. The advent of efficient quantisation schemes — first
the 4-bit GPTQ format, later the GGML and GGUF formats popularised by the
llama.cpp project — flipped that assumption. A model with several billion
parameters can now be packed into a few gigabytes on disk, loaded into
ordinary system memory, and queried in real time on a laptop, a small
single-board computer, or even a USB stick. The trade-offs of this approach
are well documented. Quantisation introduces small but measurable accuracy
losses; CPU inference is slower than GPU inference; and very large models
remain out of reach for low-end hardware. But the simplicity of the
deployment model has made local inference a practical option for an
increasingly wide range of use cases, including offline assistants, air-
gapped tooling, classroom demonstrations, and privacy-sensitive workflows
where data should never leave the user's machine.

Section two. The llama.cpp runtime.

llama.cpp is a from-scratch C++ implementation of inference for transformer
models, originally written by Georgi Gerganov in early 2023. Its initial
goal was modest: to load and run the LLaMA model on Apple Silicon. The
project quickly grew to support dozens of model architectures, multiple
quantisation formats, and a long list of acceleration backends. As of this
writing, those backends include AVX2 and AVX-512 on x86 CPUs, NEON and
SVE on ARM CPUs, the Apple Metal framework for Apple GPUs, NVIDIA CUDA,
the cross-vendor Vulkan and SYCL APIs, and several others. The runtime is
structured around a small core (ggml) that handles tensor operations and a
larger module (llama) that handles tokenisation, sampling, and the
high-level inference loop. Models are stored on disk in the GGUF
container, a single-file format designed to embed metadata alongside
quantised weights so that the runtime can configure itself without an
external manifest.

Section three. The KV cache.

A transformer language model performs attention by comparing the current
token's query vector against the keys and values of every previous token
in the context window. To avoid recomputing those keys and values from
scratch on every step of generation, runtimes maintain a key-value cache,
or KV cache, in memory. The cache is typically stored in the same
precision as the model's working state — by default, that is a 16-bit
float, or f16. For long contexts and large models the cache can become a
significant memory consumer. A 4-billion-parameter model with an 8192-token
context can easily allocate several hundred megabytes for the KV cache
alone. To address this, llama.cpp added support for quantising the KV
cache itself. The standard quantised cache types include q8_0 (8-bit), q5_1
and q5_0 (5-bit), iq4_nl (4-bit with non-linear scaling), and several
others. Each type trades off cache size against attention quality.

Section four. The rotorquant family.

A community fork of llama.cpp, hosted at johndpope/llama-cpp-turboquant,
adds three additional cache-type families specifically tuned for KV cache
storage: turbo, planar, and iso. The default Figment build uses the
turbo3 family, which stores keys and values in a structured 3-bit
representation that preserves the dominant directions of the cache while
discarding fine-grained information. Empirically, turbo3 produces roughly
half the cache size of q8_0 with quality degradation that is barely
perceptible on most prompts. The planar3 and iso3 families apply
different rotation strategies and are exposed via the FIGMENT_KV_ROTATION
environment variable for users who want to compare them on their own
workloads.

Section five. Benchmark methodology.

The benchmark harness in this repository measures four numbers per
configuration. The first is prompt-evaluation throughput, expressed in
tokens per second, which captures how quickly the runtime can ingest a
fixed prompt. The second is token-generation throughput, also in tokens
per second, which captures how quickly new tokens can be produced once
the prompt has been processed. The third is peak resident set size,
measured in mebibytes via the GNU time utility, which captures the total
memory footprint of the process at its high-water mark. The fourth is the
KV cache size in mebibytes, parsed from the runtime's startup log. The
prompt used for these measurements is the file you are currently reading.

Section six. Reproducibility caveats.

Numbers produced by this benchmark are highly sensitive to the host CPU,
the host GPU (if any), the system memory bandwidth, the kernel scheduler,
and the specific llama.cpp build. They are useful as relative comparisons
between cache-type configurations on the same machine, but they are not
useful as absolute predictions of performance on a different machine.
Always include the machine description and date alongside the numbers
when sharing them, and always commit the raw JSON output so that future
readers can compare against a known baseline.
